# 异环抽卡模拟器 网页版 - 生产计划

> **文档版本**: V1.0
> **创建日期**: 2026-06-19
> **源项目**: NTEGachaSimulator-main (Python/Tkinter 桌面版 V0.2.1)
> **目标平台**: Web 浏览器（PC + 移动端自适应）

***

## 一、项目概述

### 1.1 项目目标

将现有的 Python/Tkinter 桌面版异环抽卡模拟器，完整移植为纯前端网页应用，部署到 Cloudflare Workers，绑定自定义域名，供所有用户通过浏览器访问。

### 1.2 核心原则

- **纯前端实现**：不依赖后端服务器，所有逻辑在浏览器端运行
- **概率保真**：严格复刻原版所有概率机制、保底系统、分支逻辑
- **体验对齐**：网页版的功能和视觉效果应与桌面版一致或更优

***

## 二、技术选型

### 2.1 前端框架

| 维度            | 选型                       | 理由                                                            |
| ------------- | ------------------------ | ------------------------------------------------------------- |
| **框架**        | 原生 HTML/CSS/JS (Vanilla) | 项目逻辑自成体系，无需 React/Vue 等重框架；Cloudflare Workers 静态资源方案对纯静态文件最友好 |
| **构建工具**      | Vite (可选)                | 开发热重载 + 生产压缩打包，输出纯静态文件                                        |
| **CSS 方案**    | 原生 CSS + CSS Variables   | 轻量，无额外依赖                                                      |
| **Canvas 渲染** | HTML5 Canvas API         | 棋盘绘制、棋子动画、骰子动画                                                |

### 2.2 部署方案

| 维度       | 方案                          | 说明                           |
| -------- | --------------------------- | ---------------------------- |
| **部署平台** | Cloudflare Pages（连接 GitHub） | 比 Workers 更适合静态站点部署，自动 CI/CD |
| **域名绑定** | Cloudflare 自定义域名            | 在 Cloudflare Dashboard 中配置   |
| **构建命令** | `npm run build` (或无构建直接托管)  | 取决于是否使用 Vite                 |

> **说明**：Cloudflare Workers 适合运行边缘计算逻辑（JS Worker），而 Cloudflare Pages 专为静态站点设计，支持 GitHub 自动部署，更适合本项目。两者都可通过 Cloudflare Dashboard 操作，用户无需命令行。如果用户坚持用 Workers，可以将静态文件通过 Workers Sites 部署。

### 2.3 随机数方案

| 方案                         | 说明                                |
| -------------------------- | --------------------------------- |
| `crypto.getRandomValues()` | 浏览器原生加密级随机数，比 `Math.random()` 更可靠 |
| 可选：seedrandom 库            | 如果需要可复现的随机序列（调试用）                 |

## 五、开发阶段划分

### 阶段一：数据层移植（核心基础）

**目标**：将 Python 的概率数据和配置完整转为 JS 常量

**任务清单**：

1. 创建 `js/data/constants.js`
   - 移植 `CELL_TYPES`（11种格子类型及概率）
   - 移植 `PITY_CONFIG`（S级保底：70抽变格、90抽硬保底）
   - 移植 `DUPLICATE_RULES`（重复获得补偿规则）
   - 移植 `SLEEP_POOL_CONFIG`（沉眠池配置）
   - 移植 `SKIN_SYSTEM`（皮肤系统配置）
   - 移植 `RECHARGE_PACKS`、`DICE_PRICE_CONFIG`（如有需要）
2. 创建 `js/data/boardData.js`
   - 移植 `CharIds` 角色 ID 常量
   - 移植 `CHARACTER_DB` 角色数据库
   - 移植 `LIMITED_BOARD_TEMPLATE` 通用规则模板
   - 移植 `BOARDS_REGISTRY` 棋盘注册表（浔 + 安魂曲）
   - 移植动态接口函数：`getBoardConfig()`, `getS_Pool()`, `getA_Pool()`, `getSkinSystem()`, `getCompanionsList()`, `getBoardRules()`
3. 创建 `js/data/boardLayout.js`
   - 移植 `get_limited_board_layout()`（浔棋盘坐标：54格主路径 + 2个9格分支）
   - 移植 `get_requiem_board_layout()`（安魂曲棋盘坐标）
   - 格子名称映射（中文格子名 → cell\_type key）

**关键注意点**：

- Python 的 `random.random()` 范围是 `[0, 1)`，JS 的 `Math.random()` 也是 `[0, 1)`，兼容
- Python 的 `random.randint(1, 6)` 包含两端，JS 需要 `Math.floor(Math.random() * 6) + 1`
- Python 字典的遍历顺序在 3.7+ 保证插入顺序，JS 的 `Object.entries()` 也保证属性顺序

### 阶段二：游戏引擎移植（核心逻辑）

**目标**：将 Python 的游戏逻辑完整移植为 JS 类

**任务清单**：

1. **`js/engine/diceManager.js`** - 骰子管理器
   - 属性：`infiniteRed = true`, `infiniteRing = true`（默认无限模式）
   - 方法：`getDiceCount()`, `consumeDice()`, `addDice()`, `canAffordRoll()`, `tryConsumeForRoll()`
2. **`js/engine/gameState.js`** - 游戏状态
   - 属性：`totalRolls`, `sPityCounter`, `giftCounter`, `isVariant`, `goldChips`, `whiteChips`, `collection`, `skinClaimed`
   - 方法：`reset()`, `addToCollection()`, `incrementS_Pity()`, `resetS_Pity()`, `checkGift()`
3. **`js/engine/rollEngine.js`** - 掷骰引擎
   - `rollDice()` → 返回 1\~6
   - `determineCellType()` → 根据 `CELL_RATE_BOUNDS` 概率分布决定格子类型
4. **`js/engine/gachaEngine.js`** - 抽卡判定引擎
   - `gachaS_Character()` → S级角色抽签（含硬保底 + 变格概率判定）
   - `gachaA_CharacterFromGift()` → 集点赠礼A级角色
   - `gachaA_Disk()` / `gachaB_Disk()` → 弧盘随机
   - `gachaCompanion()` → 于此同行
   - `gachaGift()` → 集点赠礼判定
   - `gachaSkin()` → 皮肤抽取
   - `calculateDuplicateReward()` → 重复补偿计算
5. **`js/engine/sleepPoolEngine.js`** - 沉眠池追击
   - `startChase()`, `chaseRound()` → 追击事件逻辑
6. **`js/engine/gameEngine.js`** - 主引擎（最重要）
   - `executeSingleRoll()` → 完整单次掷骰流程
   - `_resolveApprenticeChestDarkbox()` → 学徒宝箱结算（S级0.2%, B级99.8%）
   - `_resolveBraveChestDarkbox()` → 勇者宝箱结算（常规S级3% / 变格S级60%）
   - `_resolveCompanionDarkbox()` → 于此同行结算（根据格子名确定角色）
   - `_resolveMistBox()` → 迷迭棋盒结算
   - `_resolveArcadeBlindSleep()` → 弧光盲盒+沉眠池组合
   - `_resolveSkinCellDarkbox()` → 皮肤格子结算
   - `_checkSkinClaims()` → 皮肤累计领取检查
   - 分支入口/出口检测逻辑
   - 保底计数 + 变格判定 + 集点赠礼检查

**关键移植注意点**：

```
原版 game_logic.py 约 1600 行，需要逐函数移植
核心是 execute_single_roll_with_target() 方法（约 200 行），需要特别仔细
分支系统（两段式移动算法）在 main.py 的动画逻辑中，需要在 UI 层实现
```

## 七、风险与挑战

| 风险                          | 影响                         | 应对方案                                      |
| --------------------------- | -------------------------- | ----------------------------------------- |
| **Canvas 性能**               | 棋盘格子多（73格），动画可能卡顿          | 使用 requestAnimationFrame，避免每帧重绘全部；只更新变化区域 |
| **棋盘坐标映射**                  | CSV 行列号到 Canvas 像素坐标需要精确计算 | 实现 `row,col → pixelX,pixelY` 映射函数，支持缩放    |
| **分支移动逻辑**                  | 原版在 Tkinter 动画中处理分支，耦合较深   | 分离：引擎只负责逻辑判定（当前位置→目标位置），UI 只负责动画渲染        |
| **概率验证**                    | 移植后概率可能有微小偏差               | 编写自动化测试，批量抽 10 万次验证统计概率                   |
| **移动端触摸**                   | 小屏幕棋盘格子可能太小                | 提供缩放功能或棋盘滚动                               |
| **Cloudflare Workers 资源限制** | 免费套餐有请求/CPU 限制             | 纯静态站点无此问题；如用 Workers Sites 需注意 KV 存储限制    |

### 8.2 Cloudflare Pages 部署（推荐）

1. 登录 <https://dash.cloudflare.com>
2. 左侧菜单 → **Workers & Pages** → **Create** → **Pages** → **Connect to Git**
3. 选择 GitHub 仓库 `NTE-Gacha-Simulator-web`
4. 构建设置：
   - **Framework preset**: `None`（或 `Vite` 如果使用 Vite）
   - **Build command**: 留空（或 `npm run build`）
   - **Build output directory**: `/`（或 `dist`）
5. 点击 **Save and Deploy**
6. 等待部署完成，获得 `xxx.pages.dev` 域名

4. 等待 SSL 证书自动签发（通常几分钟）

- 推送到 GitHub `main` 分支后，Cloudflare Pages 会自动重新部署
- 无需手动操作
